 修复排行榜查询性能问题                                                                                                 
 Context
                                     )
 排行榜页面（RankingView）加载企业排名时存在三个问题：
 1. 硬编码查询100家企业（RankingViewModel.cs:145），超过100家的企业会被遗漏
 2. N+1查询：对每个企业发起2次计数查询，100家企业产生201次数据库请求
 3. 全量拉取代替COUNT：计数方法先查出所有匹配记录再在内存里 .Count()，数据量大时极慢

 Approach

 采用内存分组聚合方案，不需要数据库改动，将排行榜加载从 2N+1 次查询优化到 3次查询。

 - 不调用 .Range()，直接返回全部结果

最新方案：
T对于上万甚至数十万的数据量，推荐方案按优先级排序：

  1. 最推荐：物化视图（Materialized View）⭐
                                                                                                                          这是大数据量排行榜的最佳方案。PostgreSQL（Supabase）支持。
                                                                                                                          CREATE MATERIALIZED VIEW company_ranking_stats AS
  SELECT                             )
      c.id,                          )
      c.company_name,
      c.industry,
      c.city,
      c.credit_code,
      COALESCE(r.review_count, 0) as review_count,
      COALESCE(cm.comment_count, 0) as comment_count
  FROM company c
  LEFT JOIN (
      SELECT company_id, COUNT(*) as review_count
      FROM review WHERE is_valid = true
      GROUP BY company_id
  ) r ON c.id = r.company_id
  LEFT JOIN (
      SELECT company_id, COUNT(*) as comment_count
      FROM company_comment WHERE is_deleted = false
      GROUP BY company_id
  ) cm ON c.id = cm.company_id
  WHERE c.is_active = true;

  CREATE INDEX idx_ranking_stats ON company_ranking_stats (review_count DESC, comment_count DESC);

  优势：查询等同于单表查询（极快）、数据预计算好了、不需要改表结构。

  刷新：定期执行 REFRESH MATERIALIZED VIEW CONCURRENTLY company_ranking_stats;（不阻塞查询）。

  适合：排行榜这种可以容忍几分钟延迟的场景。


   物化视图 + 排序分页                                                                                                   
  物化视图本质上就是一张预计算的表，可以像普通表一样查询、排序、分页。                                                  
  创建（含索引）

      GROUP BY company_id

  WHERE c.is_active = true;

  -- 为排序字段建索引，加速分页查询
  CREATE INDEX idx_ranking_review ON company_ranking_stats (review_count DESC);
  CREATE INDEX idx_ranking_comment ON company_ranking_stats (comment_count DESC);

  C# 端查询

  // 按点评数排序 + 分页（第2页，每页20条）
  var response = await _supabaseClient
      .From<CompanyRankingStats>()
      .Order(x => x.ReviewCount, Ordering.Descending)
      .Range(20, 39)  // (page-1)*pageSize ~ page*pageSize-1
      .Get();

  但有个重要前提

  物化视图是快照数据，创建时的 ORDER BY 不会自动保留查询顺序，查询时必须显式写 ORDER BY。

  -- ❌ 这样不行，顺序不保证
  SELECT * FROM company_ranking_stats LIMIT 20 OFFSET 0;

  -- ✅ 必须显式排序
  SELECT * FROM company_ranking_stats
  ORDER BY review_count DESC
  LIMIT 20 OFFSET 0;

  关于刷新

  物化视图需要手动刷新：

  -- 后台定时刷新（比如每5分钟一次，或数据变更时触发）
  REFRESH MATERIALIZED VIEW CONCURRENTLY company_ranking_stats;

  如果排行榜需要接近实时，可以考虑：
  - 触发器 + 冗余字段：在 insert/update review/comment 时自动更新 company 表的计数
  - 或应用层双重策略：物化视图做主查询，Redis/MemoryCache 做热点数据缓存

  总结

  ┌──────────────┬────────────────────────────────┐
  │     需求     │        物化视图能否满足        │
  ├──────────────┼────────────────────────────────┤
  │ 全部数据排序 │ ✅ 可以，查询时写 ORDER BY     │
  ├──────────────┼────────────────────────────────┤
  │ 分页         │ ✅ 可以，用 Range/Offset+Limit │
  ├──────────────┼────────────────────────────────┤
  │ 排序性能     │ ✅ 有索引很快，和普通表一样    │
  ├──────────────┼────────────────────────────────┤
  │ 实时性       │ ⚠️需要定时刷新，有几分钟延迟  │
  ├──────────────┼────────────────────────────────┤
  │ 数十万数据   │ ✅ 没问题，查询极快            │
  └──────────────┴────────────────────────────────┘

  对于你上十万数据的场景，物化视图 + 索引 是目前最平衡的方案。



其他方案：应用层做缓存，或者排行榜不查全部数据等等
